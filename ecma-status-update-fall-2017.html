JavaScript Proposals Status Update: Fall 2017

I thought I might start doing quarterly updates on the status of ECMAScript (the JavaScript standard) proposals.

1. Regular Expressions are on fire

There are currently five(!!) active proposals to improve regular expressions, all of which are currently at Stage 3. There is also the new matchAll API currently at Stage 1.

RegExp Lookbehind Assertions

Stage 3 Proposal by Gorkem Yakin, Nozomu Katō, and Daniel Ehrenberg

One of the lesser used, but extremely powerful features of JavaScript's regular expressions is "lookahead assertions". That is, the ability to assert that a sequence of characters is followed by another sequence of characters.

For example, if you want to match a number followed by a percentage sign, you could write the regex: 

/\d+(?=%)/.exec("123 or 456%") // ["456"]

Alternatively, if you want to match a number that is not followed by a percentage sign, you could write: 

/\d+(?!%)/.exec("123 or 456%") // ["123"]

However, (until this proposal is standardized) we don't have a way to do the inverse operations, "lookbehind assertions". Or rather, to match a sequence of characters that is lead by another sequence of characters.

This proposal adds the syntaxes (?<=...) and (?<!...) to regular expressions. So if you wanted to match a number lead by a dollar sign, you could write the regex: 

/(?<=\$)\d+/.exec("123 or $456") // ["456"]

And if you wanted to match any number not lead by a dollar sign you could write: 

/(?<!\$)\d+/.exec("123 or $456") // ["123"]

Status

There are currently no issues raised with this proposal and it is expected to move forward to Stage 4 pending engine implementations.

Due to the difficulty involved in compiling lookbehind assertions into something that works today, there is no Babel plugin to enable this feature at this time.

RegExp Unicode Property Escapes

Stage 3 Proposal by Mathias Bynens, Brian Terlson, and Daniel Ehrenberg

If you wanted to write a regular expression that matches every Greek character, you'd have a tough time. Your only option would be to get a list of them all and specify every single one of them manually. 

/[\u0370-\u0373\u0375-\u0377\u037A-\u037D\u037F\u0384\u0386.../

You'd also have to keep track of when the Unicode standard updates, because you'll need to keep this list up to date. Unless your name is Mathias Bynens, you're probably not going to do that.

Instead, with this proposal, you can specify a character set using the properties that the Unicode standard provides. Using the syntax: 

/\p{UnicodePropertyName=UnicodePropertyValue}/u
  1. /u unicode regex flag
  2. \p{...} unicode property escapes

So if you want to match any greek character: 

/\p{Script=Greek}/u.test('π'); // true

There's all sorts of properties you can use, from \p{White_Space} to \p{Emoji}, you can see a list of them in the proposal.

Status

There are currently no major issues raised with this proposal and it is expected to move forward to Stage 4 pending engine implementations.

The proposal includes two methods that you might use in order to work with unicode properties today. At this time there is no Babel plugin implementing this feature.

RegExp named capture groups

Stage 3 Proposal by Daniel Ehrenberg and Brian Terlson

Imagine you have this regex for matching a date: 

/(\d{4})-(\d{2})-(\d{2})/.exec('2017-12-01');
// result = ['2017-12-01', '2017', '12', '01']

Each (capture) group is assigned to an index in the returned array. The problem with this is that if you change your regex, these indexes can also change, and you'll have to keep your code in sync.

Instead, if you could use the new named capture group proposal. You can write the regex like this: 

/(?<year>\d{4})-(?<month>\d{2})-(?<day>\d{2})/u.exec('2017-12-01');
// result = [...]
// result.groups = { year: '2017', month: '12', day: '01' }]

Getting back named properties is much easier to use and maintain.

These named properties can then be used in replacement targets. 

'2017-12-01'.replace(dateRegex, '$<day>/$<month>/$<year>');
// result = '01/12/2017'

You can even back reference named properties in your regex. 

let duplicate = /^(?<half>.*).\k<half>$/u;
duplicate.test('a*b'); // false
duplicate.test('a*a'); // true

Status

There are currently no major issues raised with this proposal and it is expected to move forward to Stage 4 pending engine implementations.

You can enable this feature today using the transform-modern-regexp Babel plugin by Dmitry Soshnikov.

s (dotAll) flag for regular expressions

Stage 3 Proposal by Mathias Bynens and Brian Terlson

In regular expressions, a . matches a single character, regardless of what character it is. But in JavaScript, there are some weird edge cases with newlines.

This proposal adds a new "dot all" s flag to fix the problem. 

/foo.bar/s.test('foo\nbar'); // true

Status

There are currently no major issues raised with this proposal and it is expected to move forward to Stage 4 pending engine implementations.

You can enable this feature today using the transform-modern-regexp Babel plugin by Dmitry Soshnikov.

Legacy RegExp features in JavaScript

Stage 3 Proposal by Mark Miller and Claude Pache

This proposal specifies the behavior of "deprecated" JavaScript RegExp APIs:

  • Static properties RegExp.$1
  • RegExp.prototype.compile

These features already exist in implementations (with inconsistencies), this proposal formalizes the "least bad thing" they should do to maintain web compatibility.

String#matchAll

Stage 1 Proposal by Jordan Harband

If you call String#match() with a global g regex with multiple capturing groups like: 

'1+2 3+4'.match(/(\d+)\+(\d+)/)
// ['1+2', '3+4']

You'll get an array with the matches back. However, these matches don't include the capture groups. In fact, it's really annoying to get all the capture groups back.

Instead, the proposed String#matchAll API would return an iterator of the matches like: 

let matches = '1+2 3+4'.matchAll(/(\d+)\+(\d+)/);

for (let match of matches) {
  // 1. match = ['1+2', '1', '2']
  // 2. match = ['3+4', '3', '4']
}

Status

There's a lot still to be done with this proposal to get it to Stage 4, it's promising, but use at your own risk. There is a shim/polyfill available on npm by Jordan Harband.

Null Coalescing

Stage 1 Proposal by Gabriel Isenberg

It's common in JavaScript to have an expression followed by a || value to provide a default. 

let value = response.value || 42;

However, using || means that any falsy value false/0/"" will get the default value. The safer alternative to this would be to check against null and undefined manually. 

let value = response.value == null ? response.value : 42;

Using this proposal, you could simplify that to this syntax. 

let value = response.value ?? 42;

Status

This proposal just made it from Stage 0 to Stage 1. It is very risky to use at this point.

Babel is tracking this proposal.
