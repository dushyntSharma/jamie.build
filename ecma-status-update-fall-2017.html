TC39 Proposals: Fall 2017 Status Update

I thought I might start doing quarterly updates on the status of ECMAScript (the JavaScript standard) proposals.

1. Regular Expressions are on fire

There are currently five(!!) active proposals to improve regular expressions, all of which are currently at Stage 3. There is also the new matchAll API currently at Stage 1.

RegExp Lookbehind Assertions

Stage 3 Proposal by Gorkem Yakin, Nozomu Katō, and Daniel Ehrenberg

One of the lesser used, but extremely powerful features of JavaScript's regular expressions is "lookahead assertions". That is, the ability to assert that a sequence of characters is followed by another sequence of characters.

For example, if you want to match a number followed by a percentage sign, you could write the regex: 

/\d+(?=%)/.exec("123 or 456%") // ["456"]

Alternatively, if you want to match a number that is not followed by a percentage sign, you could write: 

/\d+(?!%)/.exec("123 or 456%") // ["123"]

However, (until this proposal is standardized) we don't have a way to do the inverse operations, "lookbehind assertions". Or rather, to match a sequence of characters that is lead by another sequence of characters.

This proposal adds the syntaxes (?<=...) and (?<!...) to regular expressions. So if you wanted to match a number lead by a dollar sign, you could write the regex: 

/(?<=\$)\d+/.exec("123 or $456") // ["456"]

And if you wanted to match any number not lead by a dollar sign you could write: 

/(?<!\$)\d+/.exec("123 or $456") // ["123"]

Status

There are currently no issues raised with this proposal and it is expected to move forward to Stage 4 pending engine implementations.

Due to the difficulty involved in compiling lookbehind assertions into something that works today, there is no Babel plugin to enable this feature at this time.

RegExp Unicode Property Escapes

Stage 3 Proposal by Mathias Bynens, Brian Terlson, and Daniel Ehrenberg

If you wanted to write a regular expression that matches every Greek character, you'd have a tough time. Your only option would be to get a list of them all and specify every single one of them manually. 

/[\u0370-\u0373\u0375-\u0377\u037A-\u037D\u037F\u0384\u0386.../

You'd also have to keep track of when the Unicode standard updates, because you'll need to keep this list up to date. Unless your name is Mathias Bynens, you're probably not going to do that.

Instead, with this proposal, you can specify a character set using the properties that the Unicode standard provides. Using the syntax: 

/\p{UnicodePropertyName=UnicodePropertyValue}/u
  1. /u unicode regex flag
  2. \p{...} unicode property escapes

So if you want to match any greek character: 

/\p{Script=Greek}/u.test('π'); // true

There's all sorts of properties you can use, from \p{White_Space} to \p{Emoji}, you can see a list of them in the proposal.

Status

There are currently no major issues raised with this proposal and it is expected to move forward to Stage 4 pending engine implementations.

The proposal includes two methods that you might use in order to work with unicode properties today. At this time there is no Babel plugin implementing this feature.

RegExp named capture groups

Stage 3 Proposal by Daniel Ehrenberg and Brian Terlson

Imagine you have this regex for matching a date: 

/(\d{4})-(\d{2})-(\d{2})/.exec('2017-12-01');
// result = ['2017-12-01', '2017', '12', '01']

Each (capture) group is assigned to an index in the returned array. The problem with this is that if you change your regex, these indexes can also change, and you'll have to keep your code in sync.

Instead, if you could use the new named capture group proposal. You can write the regex like this: 

/(?<year>\d{4})-(?<month>\d{2})-(?<day>\d{2})/u.exec('2017-12-01');
// result = [...]
// result.groups = { year: '2017', month: '12', day: '01' }]

Getting back named properties is much easier to use and maintain.

These named properties can then be used in replacement targets. 

'2017-12-01'.replace(dateRegex, '$<day>/$<month>/$<year>');
// result = '01/12/2017'

You can even back reference named properties in your regex. 

let duplicate = /^(?<half>.*).\k<half>$/u;
duplicate.test('a*b'); // false
duplicate.test('a*a'); // true

Status

There are currently no major issues raised with this proposal and it is expected to move forward to Stage 4 pending engine implementations.

You can enable this feature today using the transform-modern-regexp Babel plugin by Dmitry Soshnikov.

s (dotAll) flag for regular expressions

Stage 3 Proposal by Mathias Bynens and Brian Terlson

In regular expressions, a . matches a single character, regardless of what character it is. But in JavaScript, there are some weird edge cases with newlines.

This proposal adds a new "dot all" s flag to fix the problem. 

/foo.bar/s.test('foo\nbar'); // true

Status

There are currently no major issues raised with this proposal and it is expected to move forward to Stage 4 pending engine implementations.

You can enable this feature today using the transform-modern-regexp Babel plugin by Dmitry Soshnikov.

Legacy RegExp features in JavaScript

Stage 3 Proposal by Mark Miller and Claude Pache

This proposal specifies the behavior of "deprecated" JavaScript RegExp APIs:

  • Static properties RegExp.$1
  • RegExp.prototype.compile

These features already exist in implementations (with inconsistencies), this proposal formalizes the "least bad thing" they should do to maintain web compatibility.

String#matchAll

Stage 2 Proposal by Jordan Harband

If you call String#match() with a global g regex with multiple capturing groups like: 

'1+2 3+4'.match(/(\d+)\+(\d+)/)
// ['1+2', '3+4']

You'll get an array with the matches back. However, these matches don't include the capture groups. In fact, it's really annoying to get all the capture groups back.

Instead, the proposed String#matchAll API would return an iterator of the matches like: 

let matches = '1+2 3+4'.matchAll(/(\d+)\+(\d+)/);

for (let match of matches) {
  // 1. match = ['1+2', '1', '2']
  // 2. match = ['3+4', '3', '4']
}

Status

There's a lot still to be done with this proposal to get it to Stage 4, but at Stage 2 it is promising. There is a shim/polyfill available on npm by Jordan Harband.

3. Intl additions moving forward

Intl.PluralRules

Stage 4 Proposal by Caridy Patiño, Eric Ferraiuolo, and Alex Sexton

Pluralization can be difficult to get right, there are many different rules in different languages. This proposal adds an API for getting a pluralization "category" for a number in a specified language. You can then use this category to correctly pluralize your translations. 

let pluralization = new Intl.PluralRules('en', {
  style: 'cardinal',
});

pluralization.select(0); // 'other'
pluralization.select(1); // 'one'
pluralization.select(2); // 'other'

Status

As of the September 2017 meeting, this proposal moved to Stage 4 and is ready to be included in the next edition of the ECMAScript.

Intl.{DateTime,Number}Format#formatToParts

Stage 4 Proposal by Zibi Braniecki and Daniel Ehrenberg

Using the Intl formatting APIs you can do things like formatting dates to a particular locale. 

let dateFormatter = Intl.DateTimeFormat('en', {
  year: 'numeric',
  month: 'long',
  day: 'numeric'
});

console.log(dateFormatter.format(new Date()));
// "September 29, 2017"

But if you wanted to do something like make the month bold, there's no supported way to do that until this proposal. 

let dateFormatter = Intl.DateTimeFormat('en', {...});
let dateFormatParts = dateFormatter.formatToParts(new Date());

let dateStr = dateFormatParts.map(part => {
  if (part.type === 'month') {
    return `<b>${part.value}</b>`;
  } else {
    return part.value;
  }
}).join('');

console.log(dateStr);
// "<b>September</b> 29, 2017"

Status

As of the September 2017 meeting, this proposal moved to Stage 4 and is ready to be included in the next edition of the ECMAScript. It is implemented in SpiderMonkey and Chrome Canary.

Intl.Segmenter

Stage 3 Proposal by Daniel Ehrenberg

Across languages there are many different rules on how characters (graphemes), words, and sentences are broken up. This proposal adds an API that allows you to iterate of the different segments of text depending on the language. 

let segmenter = Intl.Segmenter("fr", {type: "word"});
let segments = segmenter.segment("Ceci n'est pas une pipe");

for (let {index, breakType} of segmentIterator) {
  console.log(`index: ${index}, breakType: ${breakType}`);
  break;
}

// index: 3, breakType: letter

Status

In the September 2017 meeting, this proposal moved forward to Stage 3. Pending implementations, this should move forward to Stage 4 in a future meeting.

Intl.RelativeTimeFormat

Stage 3 Proposal by Zibi Braniecki and Daniel Ehrenberg

It's very common to want to display dates and times relatively to the current time. Saying something happened "yesterday" can be much nicer than saying "September 28, 2017".

Many people use libraries like MomentJS for this purpose. But with this proposal, it will be built into JavaScript. 

let formatter = new Intl.RelativeTimeFormat("en", {
  style: "long",
});

formatter.format(-1, "day");
// "yesterday"

Status

In the September 2017 meeting, this proposal moved forward to Stage 3. Pending implementations, this should move forward to Stage 4 in a future meeting.

4. New/Interesting stage 1 proposals

Optional Chaining

Stage 1 Proposal by Gabriel Isenberg

When looking for a deeply nested property in an object, you often need to check if parent properties exist. 

let street = user.address && user.address.street;

And lots of APIs will either return an object or null/undefined, and to get a property you need to check that it is defined. 

let fooInput = myForm.querySelector('input[name=foo]');
let fooValue = fooInput ? fooInput.value : undefined;

This proposal adds a new operator which allows you to handle these cases much easier: 

let street = user.address?.street;
let fooValue = myForm.querySelector('input[name=foo]')?.value;

There's even a form that helps you when calling methods that are only sometimes defined. 

iterator.return?.() // manually close an iterator
// skip the test in older web browsers
if (myForm.checkValidity?.() === false) {
  // form validation fails
  return;
}

Status

At Stage 1, this proposal still has a lot of details to work out, and very well may change drastically or not make it into the standard at all. In July it was presented to TC39, seeking Stage 2, and did not reach consensus to move forward. So use at your own risk.

There is a Babel 7 plugin to try this feature out today if you are using Babel 7 (which is in beta).

Null Coalescing

Stage 1 Proposal by Gabriel Isenberg

It's common in JavaScript to have an expression followed by a || value to provide a default. 

let value = response.value || 42;

However, using || means that any falsy value false/0/"" will get the default value. The safer alternative to this would be to check against null and undefined manually. 

let value = response.value == null ? response.value : 42;

Using this proposal, you could simplify that to this syntax. 

let value = response.value ?? 42;

Status

This proposal just made it from Stage 0 to Stage 1. It is very risky to use at this point.

Babel is tracking this proposal.

Pipeline Operator

Stage 1 Proposal by Daniel Ehrenberg

Composing functions can be tedious in JavaScript. Your choices are either wrapping function calls with more functions calls, or creating temporary variables to unravel them. 

let result = generate(transform(parse(tokenize(code))));
let tokens = tokenize(code);
let ast = parse(code);
let transformed = transform(ast);
let result = generate(transformed);

Instead, with this proposal, you could use the pipeline operator to pass the result of calling one function into the next. 

let result = tokenize(code) |> parse |> transform |> generate;

Status

This proposal is currently at Stage 1 and is popular within the community. However, there are a lot of questions about it's design that it will have to answer in order to make it further in the standardization process. Use this syntax at your own risk.

There are some questions about how this proposal relates to the Bind Operator proposal which is solving overlapping problems. It is likely that only one of the two will be accepted.

Babel is tracking this proposal. Babylon is able to parse the syntax, and there is a third-party transform to enable it.

Partial Application Syntax

Stage 1 Proposal by Ron Buckton

One of the major limitations to the pipeline operator is how it is designed around single argument functions. If you want to pass multiple arguments you need to create additional functions. 

let result = tokenize(code, tokenizeOpts)
  |> tokens => parse(tokens, parseOpts)
  |> ast => transform(ast, transformOpts)
  |> transformed => generate(transformed, generateOpts);

If we instead had a way to do partial application, which this proposal adds to the language, we could avoid having to create additional functions. 

let result = tokenize(code, tokenizeOpts)
  |> parse(?, parseOpts)
  |> transform(?, transformOpts)
  |> generate(?, generateOpts);

Also in this proposal is the ability to have a non-fixed number of arguments placeholder. 

const maxGreaterThanZero = Math.max(0, ...);
maxGreaterThanZero(1, 2); // 2
maxGreaterThanZero(-1, -2); // 0

Status

As of the September 2017 meeting, this proposal sits at Stage 1. But there was some hesitation to even give it that because of concerns about the syntax. It is likely that if this proposal moves forward it will do so with a much different syntax.

The non-fixed "spread" placeholder seems to be particularly conversial as it adds a lot of complexity to the writing and teaching of the spec. It could also be easily confused with ...spreading arguments into a function. 

fn(...b); // spread args, calls fn
fn(...,b); // placeholder, does not call fn

Babel is not yet tracking this proposal.

5. Promises

If you're looking for more of the Bluebird/etc extensions to Promises, Jordan Harband is your new best friend. He's been pushing through two proposals to standardize some of the more popular extensions.

Promise.prototype.finally

Stage 3 Proposal by Jordan Harband

If you want to add a step to a promise chain which responds whether or not there was an error. You currently need to write it like this. 

function respondAlways() { ... }
promise.then(respondAlways, respondAlways)

Even then you still have the problem that you are catching errors when you probably don't want to. In effect, we have promise equivalents to try/catch but not finally.

This proposal adds a .finally method that reflects the behavior of finally {}. 

promise.finally(() => { ... })

Status

This proposal is currently at Stage 3 and is preparing to move to Stage 4 in a future meeting.
